# 微信群全局画像与群记忆全量重构执行计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development`（推荐）或 `superpowers:executing-plans` 按任务逐步实施。本文使用 `- [ ]` 复选框跟踪进度。

**Goal:** 将当前“按群隔离群友画像 + scoped memory + candidate 审核”的旧微信群知识链路，重构为“同一微信号唯一全局画像 + 按群隔离群记忆 + 无审核直接学习写入”的新架构，并废弃旧画像与旧群记忆数据。

**Architecture:** 复用现有 `WechatGroupArchive` 作为消息事实源，新增独立的画像存储、群记忆存储、学习运行状态与上下文装配服务。运行时只注入当前群的群记忆和当前发言人/被提及成员的全局画像；学习链路只消费已归档且按顺序推进的消息前缀，不再走候选审核流。

**Tech Stack:** Python、SQLite、现有 `channel/wechat_group/` 渠道链路、Web 控制台、`unittest`

---

## 前置约束

1. 已确认业务决策
   - 同一个 `sender_id` 只保留一份全局画像。
   - 成员画像不再做 `pending / approve / reject` 审核。
   - 旧成员画像、旧 revision、旧 candidate、旧 scoped group memory 数据全部废弃，不迁移。
   - 群记忆也同步全量重构，不沿用当前 scoped memory chunk 体系。
2. 本次实施不做的事
   - 不改 Wechaty sidecar 协议。
   - 不新增桌面端 `desktop/` UI，默认只改 Web 控制台。
   - 不接入新的向量库、Embedding Provider、外部数据库。
3. 设计来源
   - 已确认高层方案文档：`plans/20260704_微信群完整重构方案.md`
   - 参考架构：`MumuBot` 的 learner 与 memory 分层思路，而不是直接照搬其语言、存储或工具链。

### 2. 画像命名记录表

```sql
CREATE TABLE IF NOT EXISTS wechat_group_profile_name_records (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    sender_id TEXT NOT NULL,
    room_id TEXT NOT NULL,
    room_name TEXT NOT NULL DEFAULT '',
    display_name TEXT NOT NULL,
    source_kind TEXT NOT NULL DEFAULT 'message',
    last_seen_at INTEGER NOT NULL DEFAULT 0,
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_wechat_group_profile_name_records_sender
ON wechat_group_profile_name_records(sender_id, updated_at, id);
```

### 3. 群记忆表

```sql
CREATE TABLE IF NOT EXISTS wechat_group_group_memories (
    memory_id TEXT PRIMARY KEY,
    room_id TEXT NOT NULL,
    content TEXT NOT NULL,
    source_kind TEXT NOT NULL DEFAULT 'learning',
    evidence_message_ids_json TEXT NOT NULL DEFAULT '[]',
    evidence_text TEXT NOT NULL DEFAULT '',
    status TEXT NOT NULL DEFAULT 'active',
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_wechat_group_group_memories_room_status
ON wechat_group_group_memories(room_id, status, updated_at);
```

### 4. 学习游标表

```sql
CREATE TABLE IF NOT EXISTS wechat_group_learning_cursors (
    room_id TEXT PRIMARY KEY,
    last_archive_row_id INTEGER NOT NULL DEFAULT 0,
    updated_at INTEGER NOT NULL
);
```

### 5. 学习运行表

```sql
CREATE TABLE IF NOT EXISTS wechat_group_learning_runs (
    run_id TEXT PRIMARY KEY,
    room_id TEXT NOT NULL,
    mode TEXT NOT NULL,
    batch_start_row_id INTEGER NOT NULL DEFAULT 0,
    batch_end_row_id INTEGER NOT NULL DEFAULT 0,
    batch_message_count INTEGER NOT NULL DEFAULT 0,
    profile_update_count INTEGER NOT NULL DEFAULT 0,
    group_memory_upsert_count INTEGER NOT NULL DEFAULT 0,
    status TEXT NOT NULL,
    failed_reason TEXT NOT NULL DEFAULT '',
    started_at INTEGER NOT NULL,
    finished_at INTEGER NOT NULL DEFAULT 0
);

CREATE INDEX IF NOT EXISTS idx_wechat_group_learning_runs_room_time
ON wechat_group_learning_runs(room_id, started_at, run_id);
```

## 风险与控制

1. 风险：全局画像会丢失群内差异语义。
   - 控制：通过 `wechat_group_profile_name_records` 保留 `room_id + display_name` 来源信息。
2. 风险：去掉向量检索后，群记忆召回精度阶段性下降。
   - 控制：先用“精确匹配 + 包含匹配 + 最近活跃”三段排序，保证稳定性优先；后续若要升级检索，再单开任务。
3. 风险：删除旧配置会影响已有设置页保存。
   - 控制：同一批次同时改 `config.py`、`config-template.json`、`web_channel.py`、`console.js`、`tests/test_wechat_group_web.py`。
4. 风险：旧数据库表仍留在本地，容易与新实现混淆。
   - 控制：代码层完全不再读取旧表；必要时在后续单独补充一次“物理清理旧表”小任务，但不放进本次主链路。
